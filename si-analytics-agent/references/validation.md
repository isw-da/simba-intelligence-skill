# Validation Playbook

Validation is how you find out which of the three failure modes is still leaking
through. Three layers: offline evals, ablation, online validation.

## Offline evaluations

Simple question / answer pairs. Like offline testing for an ML model: they do not
tell you online performance, but they reveal critical gaps.

Two kinds:
- Dashboard-based evals: auto-generated by Claude then human-validated, covering
  the most common stakeholder questions.
- Long-tail evals: feed Claude business context (roadmaps, table docs) and have it
  generate plausible questions across the rest of the domain.

Continuously harvest every stakeholder correction in a thread as a candidate eval.

Best practices:
- Anchor ground truth so it cannot drift. Pin every eval to a snapshot date, write
  against a stable fact table, or grade the agent's query rather than its number.
  Wire the suite into CI so a PR touching a dependency re-runs the affected evals.
- Store results like telemetry, not test logs. Each run lands in a warehouse table
  with skill version, git SHA, model ID, per-assertion pass/fail, token count and
  wall-clock. "Did that change help?" becomes a query.
- Gate launches per domain. A domain owner cannot announce the agent to
  stakeholders until their eval slice clears a threshold (start around 90 percent).
- Calibrate the number of evals. Diminishing returns past a few dozen per topic,
  and the ceiling drops with each new model generation.
- Offline eval accuracy should approach 100 percent, and every correct answer
  should hit the semantic layer where one exists.

## Ablation techniques

Hold the offline eval set fixed, vary exactly one component, compare pass rates.
Each run takes about an hour and replaces a lot of arguing.

- Design for null results. Anthropic's most useful ablation was negative: giving
  the agent grep access to thousands of prior SQL files moved accuracy by less than
  a point. The information was present and read, and still not used. The bottleneck
  was structure (mapping a question to the right entity), not access. Do not
  over-invest in raw retrieval over curated routing.
- Ablate at PR granularity. Every meaningful skill edit gets a before/after run on
  the relevant eval slice, with the delta in the PR description.
- Keep a short list of what did not work, so the next person does not re-run it.

## Online validation

- Adversarial review: a skill that aggressively challenges the final answer's
  assumptions. About +6 percent accuracy, at roughly +32 percent tokens and
  +72 percent latency. Worth it for high-stakes answers.
- Provenance footer: every response carries source tier, freshness and owner. It
  does not make the answer more correct but lets the consumer judge trust. A
  "raw table, freshness unknown" footer is a signal to verify before forwarding.
- Data quality checks: confirm the referenced field is up to date, complete and
  free of anomalies.
- Passive monitoring: track the share of queries resolving through the semantic
  layer, and the share of responses drawing correction language. Review weekly
  alongside the offline pass rate.
- Active correction harvesting: a scheduled agent scans stakeholder channels for
  correction language, drafts a one-line fix to the relevant reference doc, and
  opens a PR tagged to the domain owner. Keep the fix path boring: edit markdown,
  merge, auto-sync. The same corrections feed back into the offline eval set.

## The failure mode none of this fully catches

The silent one: the answer is wrong but plausible and used without objection.
Mitigations are the provenance footer, explicit human sign-off on anything
leadership-bound, and a standing eval for each domain's top KPIs that sanity-checks
against the blessed dashboard daily. There is no robust full solution yet.
